\section*{Introduction}

Population genetics relies on a rich toolkit of summary statistics to characterize
genetic variation within and between populations. Statistics such as nucleotide
diversity ($\pi$), Tajima's $D$, $F_{ST}$, and linkage disequilibrium (LD) are
foundational to studies of demographic history, natural selection, population
structure, and admixture. As whole-genome sequencing datasets have grown from
hundreds to hundreds of thousands of individuals---exemplified by projects such
as the Anopheles 1000 Genomes \citep{ag1000g2017} and UK Biobank
\citep{bycroft2018}---the computational cost of these analyses has become a
serious bottleneck.

A number of software tools have been developed to compute population genetics
statistics from genomic data. VCFtools \citep{danecek2011} was among the first
general-purpose tools, providing command-line access to diversity, $F_{ST}$,
and LD calculations from VCF files, but was designed for the sample sizes of
the early 1000 Genomes era. \texttt{scikit-allel} \citep{miles2024} brought
these analyses into the Python ecosystem with NumPy-based array operations,
offering a broader repertoire of statistics including haplotype-based selection
scans and $F$-statistics; however, it is no longer under active development.
\texttt{pixy} \citep{korunes2021} addressed the important problem of missing
data bias in estimates of $\pi$, $d_{XY}$, and $F_{ST}$, but is limited to
those three statistics. At the other end of the spectrum, PLINK
\citep{purcell2007, chang2015} and ANGSD \citep{korneliussen2014} are
high-performance C/C++ tools but are oriented toward GWAS and genotype
likelihood workflows, respectively, rather than the summary statistics used in
evolutionary population genetics. PopGenome \citep{pfeifer2014} offered a
comprehensive R-based toolkit but has been removed from CRAN and is no longer
maintained. For tree sequence data, tskit \citep{jeffery2026} provides
exceptionally efficient computation of a general class of statistics, but
requires data in tree sequence format rather than the VCF and Zarr formats
that remain the standard output of variant calling pipelines. Across all of
these tools, computation is performed on the CPU and statistics are typically
computed one at a time across genomic windows. For large-scale projects
involving windowed genome scans of multiple statistics across many populations,
wall-clock times of hours to days are common. This limitation is particularly
acute for two-locus statistics such as LD, where pairwise computations over
variant sites are inherently $O(n^2)$ and form a bottleneck in inference
pipelines such as \texttt{moments} \citep{jouganous2017}.

Modern graphics processing units (GPUs) offer massive parallelism that is
well-suited to the data-parallel structure of population genetics computation.
Many statistics reduce to operations over matrices of haplotypes or genotypes
that can be decomposed into independent per-site or per-window calculations,
making them natural candidates for GPU acceleration. While GPUs have been
applied to specific problems in population genetics---notably forward-in-time
simulation \citep{gofish2017} and demographic inference via diffusion
approximations \citep{gutenkunst2021}---no general-purpose GPU-accelerated
library for computing population genetics summary statistics from empirical
data has been available.

Here we present \pggpu, a Python library that leverages NVIDIA GPUs via CuPy
\citep{cupy2017} to compute a comprehensive catalog of population genetics
statistics. \pggpu implements over 70 functions spanning diversity, divergence,
the site frequency spectrum, linkage disequilibrium, haplotype-based selection
scans, admixture statistics, PCA, and relatedness---covering the large majority
of statistics in routine use. The library introduces fused CUDA kernels that
compute multiple statistics in a single pass over the data, and implements the
generalized theta framework of \citet{achaz2009} to efficiently derive all
standard diversity and neutrality test statistics from a single site frequency
spectrum computation. \pggpu provides a drop-in replacement for the LD parsing
module of \texttt{moments}, achieving up to 275-fold speedups while maintaining
numerical accuracy at machine precision. Across windowed genome scans, \pggpu
delivers a median $139$-fold and up to $1{,}096$-fold speedup relative to
\texttt{scikit-allel}. \pggpu is
open source and freely available at
\url{https://github.com/kr-colab/pg_gpu}.

\section*{Design and Implementation}

\pggpu is implemented in Python on top of the CuPy NVIDIA GPU
array library \citep{cupy2017}, with around 50 custom CUDA kernels
written as \texttt{cupy.RawKernel} extensions and dense linear-algebra
operations dispatched to cuBLAS via CuPy. The package depends only on
CuPy, NumPy, SciPy, scikit-allel, msprime, and zarr, and is
distributed via a \texttt{pixi} environment that pins a CUDA 12 build
of CuPy together with all Python dependencies, so a complete
analysis stack can be installed with a single command.

\subsection*{Core data abstractions}

Two thin wrapper classes carry data through every analysis path. The
\texttt{HaplotypeMatrix} stores phased haplotypes as an
$n_{\mathrm{hap}}\times n_{\mathrm{var}}$ int8 array (with $-1$
denoting missing genotypes), variant positions, and optional named
sample sets. The companion \texttt{GenotypeMatrix} stores unphased
diploid genotypes in a $n_{\mathrm{var}}\times n_{\mathrm{samp}}$
uint8 layout for kernels whose arithmetic naturally factorizes over
the genotype tabulation $(n_{AA}, n_{Aa}, n_{aa})$. Both classes
retain a single canonical buffer per object and route between CPU
(NumPy) and GPU (CuPy) backings transparently via a \texttt{device}
property, so the same user-facing call works whether the caller has
a tree-sequence (\texttt{HaplotypeMatrix.from\_ts}), a Zarr store
(\texttt{from\_zarr}, supporting both bio2zarr/VCZ and scikit-allel
layouts), or an in-memory NumPy array.

\subsection*{Fused kernels}

Most population-genetics statistics decompose into per-site or
per-pair reductions whose CPU implementations are bandwidth-bound:
each pass over the haplotype matrix reads tens of gigabytes of memory
to emit a handful of summary scalars. \pggpu exploits this by writing
\emph{fused} kernels that read each input element once and emit
\emph{any subset} of a related family of statistics from registers
in a single launch. The user-facing entry point
\texttt{windowed\_analysis} accepts a Python list of statistic names
and dispatches the request to the smallest set of fused kernels that
covers it: a single-population kernel produces any combination of
$\pi$, $\theta_W$, $\theta_H$, $\theta_L$, Tajima's $D$, Fay \& Wu's
$H$ (raw and normalized), Zeng's $E$, the maximum DAF estimator,
segregating-site counts, and singleton counts; a two-population
kernel additionally produces Hudson's $F_{ST}$, Weir--Cockerham
$F_{ST}$, $d_{XY}$, and $d_a$. Statistics that depend on
haplotype-pattern hashing (Garud's $H_1$, $H_{12}$, $H_{123}$,
$H_2/H_1$) and scatter-add reductions (mean nSL, DAF histogram,
$\mu_\text{SFS}$, per-window $Z_{nS}$ and $\omega$, $\mu_{\mathrm{LD}}$,
SNP-distance moments) reuse the same window-binning infrastructure
through dedicated kernel families. Two-locus statistics ($D^2$,
$Dz$, $\pi_2$, $r^2$, $D'$, Pearson $r$) share a count-based kernel
that takes the four haplotype-pair counts $(n_{AB}, n_{Ab}, n_{aB},
n_{ab})$ and the sample-size vector and emits any subset of the
scalar formulas in double precision in a single thread per output.

\subsection*{Memory-aware chunking}

A naive single-launch fused kernel materialises a transposed copy of
the matrix in a working dtype (typically int8 or float32), which can
exceed GPU memory at biobank scales. \pggpu computes a chunk size at
runtime from the free-memory query \texttt{cp.cuda.Device().mem\_info}
and dispatches to a chunked variant when the projected working set
exceeds 70\,\% of free memory. The chunked path tiles the variant
axis, calls the same kernels per chunk, and accumulates partial sums
to the final result---numerics are bit-identical to the single-shot
path. This was sufficient to keep the full 10.9-million-variant
Ag1000G arm in a single A100 GPU.

\subsection*{Missing data, accessibility, and span normalization}

Per-site valid-sample counts are threaded through every diversity,
divergence, and SFS estimator, so missingness rates that vary along
the genome do not bias the result. An \texttt{AccessibleMask}
boolean array attached to the matrix gives a lazy filtered view of
the haplotypes at accessible positions only, and the inferred total
callable-bases count is propagated automatically as the denominator
for per-base normalization. The mask is non-destructive: changing
or removing it never invalidates the original buffer.

\subsection*{Generalized $\theta$ framework}

Single-population diversity estimators and neutrality tests are
implemented as instances of the general weighted-SFS estimator
$\hat\theta = \sum_{\xi} w_\xi \xi p_\xi$
\citep{achaz2009}. \pggpu computes the SFS once per population
and obtains $\pi$, $\theta_W$, $\theta_H$, $\theta_L$, the maximum
DAF estimator and so on by re-weighting; the variance of any
linear combination of these estimators is computed from the
Achaz (2009) Eq.~9 $\alpha_n, \beta_n$ coefficients with cached
weight pairs, so Tajima's $D$, Fay \& Wu's $H^*$, and Zeng's $E$
share the same null distribution under arbitrary missing-data
patterns.

\subsection*{Interfacing with downstream tools}

\pggpu provides a drop-in replacement for the LD-parsing module of
\texttt{moments} \citep{jouganous2017}: the
\texttt{compute\_ld\_statistics} function consumes the same VCF /
recombination-map / population-file inputs and returns a dictionary
in moments' canonical \texttt{(bins, sums, stats, pops)} schema, so
existing \texttt{moments.LD.Parsing.bootstrap\_data},
\texttt{optimize\_log\_lbfgsb}, and Godambe pipelines accept the
output unchanged. Local-PCA / lostruct \citep{li2019} is implemented
as a single batched-Gram eigendecomposition over windowed views of
the matrix, followed by MDS on inter-window distances; corner
identification uses Welzl's minimum-enclosing-circle algorithm
\citep{welzl1991}.

The user-facing API mirrors the table organization: each category
corresponds to a Python module
(\texttt{pg\_gpu.diversity}, \texttt{pg\_gpu.divergence},
\texttt{pg\_gpu.selection}, and so on), and each statistic is a function
that takes a \texttt{HaplotypeMatrix} or \texttt{GenotypeMatrix} as its
first argument and returns a NumPy array. Population labels assigned
once via \texttt{load\_pop\_file} are accepted by every statistic that
needs them, so \texttt{divergence.fst\_hudson(h, "pop1", "pop2")} and
\texttt{diversity.tajimas\_d(h, population="pop1")} share the same
calling convention. Where a statistic is defined for both phased and
diploid input, the same call dispatches on container type:
\texttt{selection.garud\_h} computes haplotype-frequency $H$ statistics
on a \texttt{HaplotypeMatrix} and the diplotype analogue on a
\texttt{GenotypeMatrix} without separate function names. All per-site
estimators handle missing genotypes via per-site valid-sample counts
by default, and an attached \texttt{AccessibleMask} restricts both
numerators and span-normalisation denominators to callable bases (see
Design and Implementation), so the same call returns calibrated rates
on raw and masked data.

The catalog covers the routine workhorses---$\pi$, $\theta_W$,
Tajima's $D$, Hudson and Weir--Cockerham $F_{ST}$, $d_{XY}$, the
unfolded and joint SFS, Patterson's $F_2$/$F_3$/$D$, iHS, nSL, and
Garud's $H$---together with several less commonly supported additions
that are useful in modern workflows. Schrider et al.~\citep{schrider2018}
distance-based two-population statistics ($Z_x$, $dd$, $dd$ rank) and
the RAiSD $\mu_{\mathrm{LD}}$ component~\citep{alachiotis2018} are
provided as ordinary functions; Patterson's F-statistics return
per-site numerators and denominators separately so that
ratio-of-sums standard errors can be computed correctly with
resampling (below). The pairwise-distance moments of
\citet{schrider2018} and the diplotype-frequency spectrum used for
sweep-classifier features are likewise exposed directly.

For users who need $\theta$ estimators or neutrality tests outside the
named catalog, the \texttt{FrequencySpectrum} class provides a
direct interface to the Achaz~\citep{achaz2009} weighted-SFS
formulation. A \texttt{FrequencySpectrum} built from a
\texttt{HaplotypeMatrix} computes the SFS once on the GPU and then
exposes three operations: \texttt{theta(weights)} evaluates any
linear estimator $\hat\theta = \sum_\xi w_\xi \xi p_\xi$ given either
a name from the built-in dictionary (\texttt{pi}, \texttt{watterson},
\texttt{theta\_h}, \texttt{theta\_l}, \texttt{eta1}, etc.) or a
user-supplied callable that returns a weight vector;
\texttt{neutrality\_test} pairs any two estimators into a normalised
$D$-style test statistic with a variance computed from the cached
$\alpha_n,\beta_n$ coefficients of \citet{achaz2009}, so a custom
neutrality test built around an unusual weighting (for example,
class-specific singletons) is one call rather than a hand-derived
variance formula; and \texttt{project} produces a hypergeometrically
sub-sampled SFS at any target sample size~\citep{gutenkunst2009},
allowing comparisons across populations with different sample sizes
and bridging to demographic-inference pipelines that operate on a
projected SFS.

Almost every statistic in Table~\ref{tab:features} is also accessible
through \texttt{windowed\_analysis}, a single entry point for genome
scans that takes a list of statistic names, a window definition (in
either base pairs or SNPs), and a list of populations, and returns a
\texttt{pandas.DataFrame} with one row per window. Internally,
\texttt{windowed\_analysis} dispatches the request to the smallest
covering set of fused CUDA kernels (Design and Implementation), so a
typical scan that asks for $\pi$, $\theta_W$, Tajima's $D$,
$F_{ST}$, $d_{XY}$, Garud's $H_{12}$, and mean nSL---seven statistics
spanning four kernel families---runs in a small constant number of
GPU launches over the matrix rather than seven independent passes.
Local PCA / lostruct~\citep{li2019} is exposed through the same
interface: requesting \texttt{statistics=["local\_pca"]} returns a
\texttt{LocalPCAResult} with per-window eigenvalues and eigenvectors,
which the companion functions \texttt{pc\_dist} and \texttt{corners}
turn into the inter-window distance matrix and outlier-window
identifications shown in Figure~\ref{fig:lostruct_ag1000g}.

Two cross-cutting features round out the user-facing surface.
General-purpose \texttt{block\_jackknife} and \texttt{block\_bootstrap}
estimators~\citep{busing1999, efron1993} produce calibrated standard
errors and confidence intervals for any scalar aggregate of windowed
output. Both accept a tuple of (numerator, denominator) block arrays
in addition to a single array, so ratio-of-sums quantities such as
Patterson's $F_3$, $D$, and $F_{ST}$ are resampled with the same block
assignments on the two sides of the ratio without manual book-keeping
on the user's part. Finally, \pggpu provides a drop-in replacement for
the LD-parsing module of \texttt{moments}~\citep{jouganous2017}: the
\texttt{compute\_ld\_statistics} function consumes the same VCF,
recombination-map, and population-file inputs and returns a dictionary
in the canonical \texttt{(bins, sums, stats, pops)} schema that
existing \texttt{moments.LD} bootstrap, optimisation, and Godambe
pipelines accept without modification, so a workflow that already
fits a multi-population demographic model with \texttt{moments} can
adopt \pggpu by changing a single import.

\section*{Results}

We organize the empirical evaluation of \pggpu around four questions:
do the GPU implementations agree numerically with established CPU
tools, how fast are they on real data, how do they scale to
biobank-sized samples and genome-wide variant counts, and can a
researcher use them in an end-to-end workflow without writing GPU
code? Figure~\ref{fig:overview} previews the answer on a single
chromosome of phased Ag1000G data: \pggpu reproduces $\pi$, an
$R_0$-style heterozygosity ratio, and Tajima's $D$ along the genome
indistinguishably from scikit-allel, and computes the same windowed
diversity statistics and LD-decay summary in a small fraction of the
time.

The remainder of the section quantifies these claims systematically.
We benchmark numerical agreement against scikit-allel and PLINK2,
report wall-clock comparisons on a full chromosome arm against
scikit-allel and on multi-population LD parsing against
\texttt{moments}, characterize runtime scaling along sample-size and
variant-count axes, and demonstrate an end-to-end whole-arm Ag1000G
workflow.

\subsection*{Numerical accuracy}

We validated \pggpu against scikit-allel using a 4\,Mb region of Ag1000G
chromosome 3L (3L:1{,}000{,}000--5{,}000{,}000; 2{,}940 haplotypes after
removing missing-data sites). For every statistic with a scikit-allel
equivalent we observe agreement to either machine precision or
floating-point roundoff bounded by the order of harmonic-sum reductions
in the underlying formula (Table~\ref{tab:accuracy}). Statistics whose
estimators reduce to exact integer counts---site-frequency spectra,
two-locus haplotype-frequency tallies (Garud's $H$), and Patterson's
$F_2$---agree exactly. Statistics that involve inverse harmonics or
unbiased corrections ($\theta_W$, Tajima's $D$) differ by at most
$\sim 5\times 10^{-4}$, consistent with reordering of double-precision
sums under different reduction trees. A directly comparable Hudson
$F_{ST}$ computation against PLINK2 on simulated msprime data agrees to
$1.7\times 10^{-8}$.

\subsection*{Performance benchmarks}

To quantify the speedup attainable on real data we benchmarked \pggpu
against scikit-allel on the entire Ag1000G Phase 3 chromosome 3R arm
(2{,}940 haplotypes, 10.9 million variants). For the 23 statistics with
a directly comparable scikit-allel implementation the median speedup is
$139\times$ and the maximum is $1{,}096\times$ for Weir--Cockerham
$F_{ST}$ (Figure~\ref{fig:speedups}). The largest gains span
divergence, diversity, and pairwise linkage-disequilibrium kernels:
$1{,}096\times$ for Weir--Cockerham $F_{ST}$, $806\times$ for observed
heterozygosity, $617\times$ for all-pairwise Rogers--Huff $r$ on a
$10{,}000$-SNP block (the full chromosome-scale all-pairs computation
is intractable for scikit-allel), $376\times$ for the inbreeding
coefficient, $316\times$ for $\pi$, and $215\times$ for Tajima's $D$.
These reflect the tight coupling between allele-count tabulation and
pairwise reduction in the fused kernels. Haplotype-based selection
scans (EHH decay, Garud's $H$, XP-EHH, iHS, nSL) realize smaller but
still substantial speedups ($8.7$--$56\times$); these statistics
already enjoy efficient C extensions inside scikit-allel and the GPU
advantage is bounded by the inherently per-haplotype work. Absolute
wall-clock comparisons and the per-statistic table are reported in
Supplementary Figure~\ref{fig:walltimes}.

A second class of benchmarks compares \pggpu against \texttt{moments}
\citep{jouganous2017} for the unique role of computing
multi-population LD statistics that are used as input to
demographic inference. On a four-population coalescent simulation
(105 LD statistics across 5 distance bins, 10 replicates) \pggpu agrees
with \texttt{moments} to a maximum relative error of
$1.8\times 10^{-11}$ while reducing the per-replicate parsing time
from $256$\,s to $0.2$\,s, a $1{,}756 \pm 410\times$ speedup
(Figure~\ref{fig:moments_ld}). This makes whole-genome LD-based
inference tractable on standard hardware.

\subsection*{Scaling}

Runtime scaling was characterized along two independent axes
(Figure~\ref{fig:scaling}). To probe sample-size scaling we held the
variant count fixed at 100{,}000 and increased the number of haplotypes
from 100 to 100{,}000 (synthetic biallelic genotypes). To probe
variant-count scaling we held sample size fixed at 200 haplotypes and
generated \texttt{msprime} simulations \citep{baumdicker2022} of
increasing sequence length, yielding
between $\sim 10^3$ and $\sim 10^6$ segregating sites with realistic
linkage disequilibrium structure. Per-site statistics ($\pi$,
$\theta_W$, Tajima's $D$, Hudson $F_{ST}$, windowed three-statistic
fused kernel) scale linearly in both axes and complete in well under a
second across the tested range. Two-locus statistics show their
expected $O(n^2)$ scaling with sample size for Garud's $H$ and a
sub-quadratic profile for $Z_{nS}$, which uses a sampled-pairs
estimator. At the largest tested configuration (100{,}000 haplotypes,
100{,}000 variants) all single-pass statistics complete in under
$10$ seconds on a single A100, demonstrating that biobank-scale
sample sizes are within reach for whole-genome scans.

variants) and variant count (right, fixed at 200 haplotypes). Both
axes are log-scaled. \texttt{windowed\_3} is a fused windowed kernel
that computes $\pi$, $\theta_W$, and Tajima's $D$ in a single pass.}
\label{fig:scaling}
\end{figure}

\subsection*{Application to Ag1000G}

To demonstrate end-to-end utility we ran a full population-genetic
genome scan on Ag1000G Phase 3 chromosome 3R, partitioning the 1{,}470
samples into West African and East African groups (200 diploids each).
The full pipeline---loading 10.9 million phased variants from Zarr,
attaching the canonical accessibility mask, transferring to GPU, and
computing 19 windowed statistics in 100\,kb windows with 10\,kb step
plus per-population SFS and joint SFS---runs in 168 seconds total wall
time, of which only 30 seconds is GPU compute (Figure~\ref{fig:scan}).
The scan recovers the expected genome-wide signatures of $\pi$
heterogeneity, negative Tajima's $D$ associated with population
expansion, and elevated Hudson $F_{ST}$ between subspecies, providing
a single-call equivalent of multi-tool pipelines that would otherwise
require hours to days of CPU time.

\begin{figure}[ht]
\centering
\includegraphics[width=\textwidth]{ag1000g_genome_scan.png}
\caption{Whole-arm scan of Ag1000G chromosome 3R using \pggpu.
Left column: genome-wide windowed statistics (100\,kb windows, 10\,kb
step) including diversity ($\pi$, $\theta_W$), neutrality tests
(Tajima's $D$, Fay \& Wu's $H^*$), divergence between West and East
African populations (Hudson $F_{ST}$, $d_{XY}$), Garud's $H_{12}$,
mean nSL, and segregating-site density. Right column: West African

\end{figure}

\subsection*{Local PCA and integration with moments}

Beyond the standard catalog of summary statistics, \pggpu makes
several recently-introduced methods practical at scale. We illustrate
two complementary use cases. First, the local-PCA / lostruct
\citep{li2019} workflow---which identifies genomic windows whose
structure deviates from the genome-wide PCA---is implemented as a
single fused windowed-PCA kernel followed by MDS on the inter-window
distance matrix. Applied to a coalescent simulation
\citep{baumdicker2022} of a hard sweep,
the lostruct embedding cleanly separates neutral, linked, and sweep
windows and the leading MDS axis tracks the localized excess of
Garud's $H_{12}$ at the sweep focal site (Figure~\ref{fig:lostruct}).
The streaming-dense local-PCA engine extends the same analysis to
real-data scale on a single GPU: applied to the West African subset
of Ag1000G Phase 3 chromosome 3R (200 phased haplotypes, 10.9 million
variants, $\sim$10{,}900 non-overlapping $1{,}000$-SNP windows), the
full Li \& Ralph pipeline---per-window local PCA, Frobenius distance
between window covariances, classical MDS, and corner detection---runs
end-to-end alongside the 32\,GB haplotype buffer on an 80\,GB A100,
because the streaming engine bounds peak GPU memory at the per-window
working set rather than holding the full $\sim10{,}900$ Gram-stack in
memory. Three distinct corner clusters are recovered, each a
contiguous block of windows in physical coordinates, and the
corner-3 region near $\sim$31.5\,Mb co-localizes with a localized peak
in the companion Garud's $H_{12}$ track on the same windows
(Figure~\ref{fig:lostruct_ag1000g}). This makes whole-genome
lostruct screens for outlier population-structure regions tractable
without restricting to subsampled SNP panels.

Second, \pggpu provides a drop-in replacement for the
\texttt{moments.LD} parsing module: a three-population isolation
model fit using \pggpu-parsed LD statistics inside the standard
\texttt{moments} optimizer recovers parameter estimates indistinguishable
from those obtained with the canonical \texttt{moments.LD} parser
(Figure~\ref{fig:moments_inference}), at the speedups documented in
Figure~\ref{fig:moments_ld}.
